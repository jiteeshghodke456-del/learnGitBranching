Design proposal · Learn Git Branching

Making the staging area something you can see

Staging currently lives only in the terminal text. This draws the working directory and staging area right on the canvas, so git add visibly moves a file into staging and a commit bundles the staged files up into a new circle.

Colors, node size, labels and font are pulled straight from LGB's code (teal nodes, spring-green main, blue-purple HEAD, Menlo). Click through the flow, or hit play.

How it reads

The rules stay simple and stay out of the way of the commit graph, which is still the star:

git add

A file slides into staging

Editing a file drops an amber chip into the working directory. git add moves it up into the staging area and turns it green.

git commit

Staged files bundle into a circle

On commit, the green chips fly over and collapse into a brand new commit node. The mental model made literal: a commit is a bundle of staged changes.

git restore

And back out again

git restore --staged drops a chip back to the working directory; git restore discards it. The panel only shows on levels that teach staging, so every existing level is untouched.

phase 2

A conflict editor

Same idea for merge conflicts: a small in-app panel showing the <<< / === / >>> hunk with "keep ours / keep theirs / keep both", so people practice resolving instead of just reading markers.
